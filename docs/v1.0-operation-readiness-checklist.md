# Gemini-A11y Agent v1.0 運用開始前チェックリスト

## 1. 目的

Gemini-A11y Agent v1.0 を Google Colab / Google Sheets / Google Drive で試験運用または実運用する前に、最低限確認すべき項目を1枚で確認するためのチェックリストです。

このチェックリストは、本体コード、fixture、`gold`、`ai-v1.0` 出力を変更するための手順ではありません。通常運用前の確認、証跡整理、未確認事項の切り分けに使います。

## 2. 前提

- v1.0の既存方針を維持する。
- 対象本文、table構造、iframe title、画像alt、fixture方針を不要に変更しない。
- 外部支援ツールは評価・証跡・開発補助に限定し、通常のv1.0本体処理へ組み込まない。
- Colab / Sheets / Drive / 実ブラウザ確認は、Codex環境外で確認する。
- 実運用用の Google Sheets ID、Drive フォルダID、APIキー、GitHub token、個人情報、非公開情報をドキュメントやログに残さない。

## 3. 実行環境チェック

- [ ] Google Colabでリポジトリを取得できる。
- [ ] リポジトリ直下でコマンドを実行できる。
- [ ] `PYTHONPATH=.` が必要な場合の実行方法を確認している。
- [ ] Python依存関係をインストールできる。
  - [ ] `google-genai`
  - [ ] `beautifulsoup4`
  - [ ] `lxml`
  - [ ] `requests`
  - [ ] `trafilatura`
  - [ ] `gspread`
- [ ] Gemini APIキーをColabシークレットまたは既存運用方法で参照できる。
- [ ] `GEMINI_MODEL_ID` の既定値または上書き値を確認している。

## 4. Google Sheetsチェック

- [ ] 処理対象一覧シートが存在する。
- [ ] A〜J列の基本列が運用手順どおりである。
- [ ] K列以降のVision関連列を使う場合、既存仕様と整合している。
- [ ] URL、ファイル名、XPath、ステータス列の意味を運用者が理解している。
- [ ] テスト対象URLを1件だけ入れた状態で試験できる。
- [ ] ステータス更新が想定どおり行われる。

## 5. Google Drive出力チェック

- [ ] 出力先Driveフォルダが存在する。
- [ ] ColabからDriveへ保存できる。
- [ ] 生成HTMLのファイル名がシート指定と一致する。
- [ ] 上書き時の扱いを理解している。
- [ ] 誤って本番成果物を上書きしない試験用フォルダがある。

## 6. Gemini APIチェック

- [ ] APIキーが取得できる。
- [ ] 使用モデルが確認済みである。
- [ ] トークン使用量と概算コストの記録先がある。
- [ ] APIキーやレスポンス全文を不要にログ出力しない。
- [ ] エラー時に原因を確認できるログが残る。

## 7. fixture / 回帰検証チェック

標準確認コマンド:

```bash
git diff --check

python tools/check_saga_city_test_fixture.py

python tools/compare_saga_city_versions.py \
  --fixture-root tests/fixtures/html/saga-city-test \
  --case sg02395-composite
```

確認項目:

- [ ] `git diff --check` が成功している。
- [ ] `python tools/check_saga_city_test_fixture.py` が成功している。
- [ ] `python tools/compare_saga_city_versions.py ...` の結果を確認している。
- [ ] `previous fixture missing` は、`ai-v0` を生成しない方針による想定内警告か確認している。
- [ ] `matches_gold` は、現在出力が期待値である `gold` と一致している件数を示すことを理解している。
- [ ] `differs_from_gold` は、現在出力が `gold` と異なるが、旧版との比較上ただちに回帰とは限らない件数を示すことを理解している。
- [ ] `regressed` は、期待値や旧版比較の観点で悪化した可能性がある件数を示すことを理解している。
- [ ] `warning` は、比較対象不足や確認上の注意など、追加確認が必要な警告件数を示すことを理解している。
- [ ] `regressed` が1件でもあれば運用開始しない。
- [ ] `gold` は安易に更新しない。

## 8. HTML出力確認チェック

- [ ] 導入文が欠落していない。
- [ ] h3 / h4 が欠落していない。
- [ ] tableにcaptionがある。
- [ ] tableにtheadがある。
- [ ] th / scope が適切に付与されている。
- [ ] iframeに適切なtitleがある。
- [ ] YouTube iframe title が既存方針どおりである。
- [ ] 画像altが不要に改変されていない。
- [ ] Menu / PageTop / footer が本文に混入していない。
- [ ] `更新：YYYY年M月D日` のような単独更新行が不要に混入していない。
- [ ] 元本文の意味が変わっていない。
- [ ] ブラウザで開いて大きな崩れがない。

## 9. 外部支援ツールに関する確認

- [ ] 通常のv1.0運用では kage / Headroom / Firecrawl / agmsg を使わない。
- [ ] 外部支援ツールは本体処理へ組み込まない。
- [ ] 外部ツール出力から `gold` や `ai-v1.0` を生成しない。
- [ ] Firecrawl Markdown / clean HTML を補正入力にしない。
- [ ] kage出力HTMLを標準入力HTMLにしない。
- [ ] HeadroomでHTML本文やfixture本文を圧縮・要約・改変しない。
- [ ] agmsg を v1.0 本体処理や Colab 標準実行に組み込まない。
- [ ] agmsg のメッセージに APIキー、GitHub token、個人情報、非公開情報を含めない。
- [ ] 外部支援ツールの状態は [`external-tools-evaluation-status.md`](external-tools-evaluation-status.md) を参照する。

## 10. Codex利用時の確認

- [ ] Codexに作業依頼する場合は [`codex-workflow.md`](codex-workflow.md) を参照する。
- [ ] 長い作業では [`codex-goal-workflow.md`](codex-goal-workflow.md) を参照し、Codex CLI では `/goal` と `/plan` を使う。
- [ ] Codexに任せてよい作業・任せない作業を確認する。
- [ ] fixture / gold / ai-v1.0 を変更する場合は、明示的な理由と確認が必要である。
- [ ] Codex環境で確認できないものは、未確認としてPR本文に明記する。
- [ ] Colab / Sheets / Drive / 実ブラウザ確認はCodex環境外で確認する。

## 11. 運用開始可否の判定

| 判定 | 条件 |
| --- | --- |
| 開始可 | fixture確認がPASS、HTML出力確認で重大な欠落なし、Drive保存・Sheets更新確認済み |
| 条件付き開始 | Codex環境外の確認が一部未実施だが、試験用フォルダ・試験用シートで限定実行する |
| 開始不可 | regressionあり、本文欠落あり、table構造破壊あり、APIキーや保存先に不備あり |
